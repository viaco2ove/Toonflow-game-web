## 角色参数卡信息 -详细版
下面给你一版可直接落地的 **完整中文说明 + JSON 格式**。目标是：

* **尽量固定字段名**
* **减少 `other / 其他` 的使用**
* **让 AI 更稳定地扮演角色**
* **兼顾前台创建简单、后台参数完整**

---

# 一、设计原则

角色卡建议分成两层：

## 1. 用户创建层

用户在创建角色时，只填写少量直观字段：

* 头像
* 角色名
* 角色设定
* 角色音色
* 台词示例

这层是给人看的，输入门槛低。

## 2. 系统参数层

系统根据“角色设定”等内容，自动生成一张 **结构化参数卡 JSON**。

这层是给 AI 和游戏逻辑用的，重点是：

* 字段固定
* 含义明确
* 尽量短文本 / 枚举 / 数组化
* 避免把大量信息塞进 `other`

---

# 二、字段设计说明

---

## A. 基础信息 `basic`

用于标识角色是谁。

### 字段

* `id`：角色唯一 ID
* `name`：角色名
* `aliases`：别名、称呼
* `gender`：性别
* `age`：年龄
* `race`：种族 / 物种 / 身份类别
* `occupation`：职业 / 社会身份
* `camp`：阵营 / 立场
* `tags`：标签，便于检索

### 作用

这部分是角色识别的基础，必须尽量固定。

---

## B. 原始设定 `source`

保留用户最初输入，方便回溯。

### 字段

* `raw_setting`：用户输入的角色设定原文
* `raw_voice_style`：用户选择的角色音色
* `raw_sample_lines`：用户输入的台词示例

### 作用

这部分不是给模型直接推理的核心字段，而是作为原始输入存档。

---

## C. 世界观信息 `world`

描述角色处于什么世界里。

### 字段

* `era`：时代背景
* `region`：地区 / 活动范围
* `faction`：所属势力 / 组织
* `background_summary`：背景简介

### 作用

控制角色说话和行为时的世界观一致性。

---

## D. 外观信息 `appearance`

不要只用一个“外貌”大文本，尽量拆细。

### 字段

* `body_type`：体型
* `height`：身高描述
* `hair`：发型 / 发色
* `eyes`：眼睛特征
* `clothing_style`：穿衣风格
* `distinctive_features`：显著特征

### 作用

便于角色描述、立绘生成、剧情引用。

---

## E. 声音信息 `voice`

不要只存“温柔”“御姐”这种词，尽量结构化。

### 字段

* `tone`：音色主类型
* `speed`：语速
* `volume`：音量倾向
* `style_keywords`：声音风格关键词

### 作用

可用于：

* TTS 映射
* 台词风格控制
* 角色说话一致性

---

## F. 性格信息 `personality`

“性格”不能只写一句话，最好拆成多个维度。

### 字段

* `core_traits`：核心性格词
* `temper`：脾气类型
* `values`：重视的价值
* `likes`：喜欢
* `dislikes`：讨厌
* `fears`：恐惧
* `weaknesses`：弱点

### 作用

这是角色稳定性的核心之一。

---

## G. 行为模式 `behavior`

角色不仅要“是什么性格”，还要知道“通常怎么表现”。

### 字段

* `speaking_style`：说话方式
* `habit_actions`：习惯动作
* `decision_style`：决策方式
* `social_attitude`：社交态度

### 作用

决定角色在互动中的表达习惯。

---

## H. 动机目标 `motivation`

这个字段非常重要，很多角色不稳定就是因为没有明确目标。

### 字段

* `short_term_goal`：短期目标
* `long_term_goal`：长期目标
* `core_desire`：核心欲望
* `taboos`：禁忌

### 作用

让 AI 知道角色为什么行动、什么不能碰。

---

## I. 关系信息 `relationships`

角色必须知道怎么对待玩家和其他人。

### 字段

* `default_attitude_to_player`：默认对玩家态度
* `important_people`：重要关系人列表

### `important_people` 子字段

* `name`：名字
* `relation`：关系
* `attitude`：态度
* `notes`：补充说明

### 作用

支持剧情、亲密度、冲突和阵营关系。

---

## J. 能力信息 `abilities`

不要只写“技能”，要支持战斗、社交、魔法、被动等能力。

### 字段

每个能力包含：

* `name`：能力名
* `type`：能力类型
* `description`：说明
* `cooldown`：冷却
* `limit`：限制条件

### 作用

后续战斗系统、事件判定、AI行为都能用。

---

## K. 物品装备 `inventory`

把“物品”和“装备”区分开。

### 字段

* `items`：普通物品
* `equipment`：装备
* `resources`：资源（如金钱、人脉、权限、情报）

### 作用

避免把所有资源都写成“背包道具”。

---

## L. 互动规则 `interaction_rules`

这是 AI 游戏最重要的稳定器之一。

### 字段

* `can_romance`：是否可攻略
* `can_betray`：是否可能背叛
* `can_join_team`：是否可入队
* `response_length_preference`：回复长度偏好
* `nsfw_level`：内容限制等级
* `knowledge_boundary`：知识边界
* `secret_level`：秘密程度

### 作用

控制角色的边界和玩法表现。

---

## M. 演出信息 `performance`

这是让角色“像角色”的关键。

### 字段

* `catchphrases`：口头禅
* `sample_lines`：代表台词
* `emotion_triggers`：情绪触发点
* `first_impression`：初见印象
* `revealed_impression`：深入了解后的印象

### 作用

决定角色演出感和记忆点。

---

## N. 扩展字段 `extension`

不建议保留一个无限制的 `other`。
推荐改成：

### 字段

* `custom_traits`：少量扩展标签
* `extra_notes`：补充说明

### 作用

只做兜底，不承载核心设定。

---

# 三、字段填写规范建议

为了让 AI 更稳定，建议生成时遵循以下规则：

## 1. 优先填固定字段

能归类的内容必须归类，不能一股脑塞到扩展字段。

## 2. 不确定就留空

不要乱编内容。未知字段可用：

* 空字符串 `""`
* 空数组 `[]`
* `null`

## 3. 数组尽量短

像 `core_traits`、`likes`、`taboos` 这类字段，建议控制在 3～5 项。

## 4. 性格词标准化

例如统一用：

* 冷静
* 冲动
* 温柔
* 傲慢
* 理智
* 多疑
* 忠诚
* 狡猾

不要一会儿“嘴硬心软”，一会儿“假装不在乎其实很温柔”，这样不利于结构化。

## 5. 音色尽量枚举化

例如：

* 少女
* 少年
* 青年女
* 青年男
* 御姐
* 成熟男
* 冷淡
* 机械
* 神秘

## 6. 目标和禁忌必须单独抽出

这两个字段对剧情稳定性极重要。

## 7. `extra_notes` 不能写核心内容

如果核心内容出现在 `extra_notes`，说明字段设计还不够完整。

# 五、子结构定义建议

---

## 1. `important_people` 结构

```json
{
  "name": "",
  "relation": "",
  "attitude": "",
  "notes": ""
}
```

### 示例

```json
{
  "name": "林川",
  "relation": "师父",
  "attitude": "尊敬但畏惧",
  "notes": "对其要求极高，从不敢违抗"
}
```

---

## 2. `abilities` 结构

```json
{
  "name": "",
  "type": "",
  "description": "",
  "cooldown": "",
  "limit": ""
}
```

### `type` 推荐枚举

* combat：战斗
* social：社交
* magic：魔法
* passive：被动
* utility：辅助
* special：特殊

### 示例

```json
{
  "name": "影缚",
  "type": "magic",
  "description": "可在短时间内束缚目标行动",
  "cooldown": "30秒",
  "limit": "需在阴影环境下发动"
}
```

---

## 3. `items / equipment / resources` 建议结构

### 简单版

可以先用字符串数组：

```json
{
  "items": ["怀表", "信件"],
  "equipment": ["长刀", "披风"],
  "resources": ["黑市情报网", "少量金币"]
}
```

### 进阶版

如果后续要做游戏数值系统，可以改成对象数组。

# 七、创建页与参数卡映射关系

你现在前台字段是：

* 头像
* 角色名
* 角色设定
* 角色音色
* 台词示例

建议映射如下：

## 前台输入

```json
{
  "avatar": "",
  "name": "",
  "role_setting": "",
  "voice_style": "",
  "sample_lines": []
}
```

## 后台生成

* `name` → `basic.name`
* `role_setting` → 拆解进 `world / personality / behavior / motivation / relationships / abilities`
* `voice_style` → `source.raw_voice_style` + `voice`
* `sample_lines` → `source.raw_sample_lines` + `performance.sample_lines`

---

# 八、最小化扩展字段的建议

下面这些内容，**一定不要放进 `extra_notes`**，必须优先固定：

* 性格核心词
* 说话风格
* 短期目标
* 长期目标
* 对玩家态度
* 禁忌
* 能力
* 装备
* 知识边界
* 秘密程度
* 代表台词

只有真的无法归类时，才进入 `extension`。

# 十、最终建议

你可以直接把这套作为 **角色参数卡 v1**：

## 必须固定的部分

* `basic`
* `world`
* `appearance`
* `voice`
* `personality`
* `behavior`
* `motivation`
* `relationships`
* `abilities`
* `inventory`
* `interaction_rules`
* `performance`

## 兜底部分

* `extension`

这样可以把绝大多数角色设定都收进固定字段里，大幅减少 `other` 的膨胀。
