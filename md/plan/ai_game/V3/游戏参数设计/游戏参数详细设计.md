# 游戏参数设计（Runtime）

## 1. 目标与范围
本设计用于补齐“角色参数卡”之外的游戏运行时参数，目标是：

1. 存储每个玩家独立的游玩进度与动态状态。
2. 把关键信息结构化，避免只依赖聊天全文导致遗忘。
3. 为 LLM 提供可控上下文装载（短/中/长期记忆）。

本设计聚焦 `game_run` 维度，不改动角色卡基础模型。

## 2. 运行时核心对象
一个玩家进入一个故事后，生成一个运行实例 `game_run`。该实例包含以下核心参数域：

1. `progress`：章节进度、场景进度、主支线任务进度。
2. `world_state`：世界动态变化（势力关系、场景状态、全局变量）。
3. `player_state`：玩家当前所处地点、身份状态、关键资源。
4. `event_pool`：已触发/待触发事件，含条件和结果。
5. `relationship_state`：玩家与主要角色关系值与关系标签变化。
6. `memory_state`：短期、中期、长期记忆索引与摘要。

## 4. 记忆分级系统
### 4.1 短期记忆（Short-Term）
用途：保证当前对话连续性。

1. 载入最近 `N` 轮消息（建议 `8~16` 轮）。
2. 保留最近一次玩家输入、最近一次系统决策、最近一次事件结果。
3. 生命周期短，超过窗口自动出队。

### 4.2 中期记忆（Mid-Term）
用途：保留关键剧情事实，避免“上下文爆炸”。

1. 存“事实条目”，不存全文。
2. 每条事实包含：`topic`、`content`、`importance`、`source_turn`、`ttl`。
3. 由记忆管理 Agent 定期抽取和合并（建议每 4~8 轮执行一次）。

### 4.3 长期记忆（Long-Term）
用途：保证世界观与角色关系稳定。

1. 存放全局设定与已确认长期变化（如阵营变化、永久伤害、世界事件）。
2. 不高频更新，采用“确认后写入”策略。
3. 与 `story_bible` 版本绑定，支持回溯与重放。

## 5. 记忆管理 Agent（Memory Manager）设计
输入：最近消息窗口 + 当前 snapshot + 上一版中期摘要。  
输出：`memory_delta`（新增/更新/删除）+ `new_mid_summary`。

执行流程：

1. 抽取事实：识别事件、关系变化、任务变化、地点变化。
2. 打分去重：按 `importance` + `recency` 去重聚合。
3. 分类入池：短期保留在窗口，中期写事实库，长期只写“确认事实”。
4. 生成索引：按 `topic`、`role_id`、`chapter_id` 建索引。
5. 产出注入包：给下一轮推理使用的最小上下文集合。

## 6. 数据库落地建议
建议在现有会话表基础上新增或补齐以下表：

1. `game_run`
字段：`id, story_id, player_id, status, current_chapter_id, current_scene_id, turn, created_at, updated_at`。

2. `game_run_snapshot`
字段：`id, game_run_id, turn, snapshot_json, created_at`。
说明：每 `N` 轮 + 关键事件强制落快照。

3. `game_run_event`
字段：`id, game_run_id, event_type, event_key, payload_json, result, turn, created_at`。

4. `game_run_memory`
字段：`id, game_run_id, memory_level, topic, content, importance, ttl_turn, source_turn, status, created_at, updated_at`。
`memory_level` 建议枚举：`short|mid|long`。

5. `game_run_memory_summary`
字段：`id, game_run_id, summary_type, summary_text, based_on_turn_start, based_on_turn_end, created_at`。

## 7. 上下文装载策略（给大模型）
每轮请求模型时按顺序装载：

1. 系统指令：玩法规则 + 安全规则。
2. 长期记忆：世界观、永久状态、关键角色关系。
3. 中期记忆：高重要事实摘要（限制条数，如 `10~20` 条）。
4. 短期记忆：最近对话窗口。
5. 当前输入：玩家本轮输入。

装载预算建议：

1. 长期记忆 `15%`。
2. 中期记忆 `35%`。
3. 短期记忆 `40%`。
4. 预留 `10%` 给系统提示与函数调用。

## 8. 与角色参数卡的联动
角色参数卡是“静态+半静态底座”，运行时参数是“动态游玩层”。

联动方式：

1. `role.parameterCardJson` 作为初始角色配置源。
2. `relationship_state`、`inventory`、`ability_cd` 等变化写入 `game_run` 层，不回写原始角色卡。
3. 如果需要“继承存档开新周目”，从 `game_run_snapshot` 派生新 run。

## 9. 更新策略（MVP）
每轮处理建议顺序：

1. 写入消息日志。
2. 执行事件判定与任务判定。
3. 生成 `state_delta`。
4. 应用 `state_delta` 到内存态。
5. 必要时写 snapshot。
6. 触发记忆管理 Agent（按轮次或事件阈值）。
7. 生成下一轮模型上下文。

## 10. MVP 先做与后做
先做（必须）：

1. `game_run + snapshot + event + memory(short/mid)` 四层落地。
2. 中期记忆抽取与摘要（可先规则 + LLM 混合）。
3. 章节进度、地点、任务、关系值四类核心参数。

后做（增强）：

1. 长期记忆自动确认机制。
2. 记忆冲突检测与自动修复。
3. 多模型协同（主剧情模型 + 记忆模型 + 事件模型）。

## 11. 交付结论
task2 的“游戏参数设计”建议采用“`snapshot + delta + 分级记忆`”路线：

1. `snapshot` 负责可回放、可恢复。
2. `delta` 负责轻量高频更新。
3. 分级记忆负责降低上下文成本并提升连续性。

这套模型可以直接对接现有角色参数卡，并支持后续安卓/网页统一运行时。
