# AI故事小游戏详细设计与规则方案

## 1. 背景与问题
当前小游戏方案只有“进入小游戏后由 AI 控制局内规则”的一句描述，这对真实运行还不够，主要有 6 个问题：

1. 没有固定状态机，模型容易把小游戏和主线剧情混着写。
2. 没有定义公开信息和隐藏信息，狼人杀这类玩法很容易泄密。
3. 没有固定动作集合，用户一句自由输入就可能把规则打穿。
4. 没有明确的暂停、恢复、退出协议，打断后容易丢状态。
5. 没有统一结算与回写白名单，奖励、惩罚和主线影响不可控。
6. 没有具体参数、公式和轮次，小游戏难以稳定复现。

本次 task4 的目标，是把 7 个小游戏补成一套“可配置、可缓存、可回放、可落库、可给 Agent 直接执行”的详细设计。

## 2. 设计目标
### 2.1 核心目标
1. 小游戏必须独立运行，不能污染主线编排。
2. 小游戏必须有固定轮次、阶段、动作、资源与结算公式。
3. 小游戏必须支持 `snapshot + delta` 持久化，打断后可恢复。
4. 奖励与惩罚只能写回允许字段，不得越权改剧情。
5. 每个小游戏都要给出 MVP 可落地版本，不做“看情况自由发挥”。

### 2.2 设计原则
1. 先做稳定的 MVP 规则，再做复杂扩展。
2. 优先离散动作，不依赖超长自由对话。
3. 公开状态和隐藏状态严格分层。
4. 随机性必须可复现，不能让模型随口编概率。
5. 单局时长尽量控制在 `3~8` 轮，避免拖慢主线。

## 3. 总体架构
### 3.1 模块拆分
小游戏层建议拆成 4 个对象：

1. `mini_game_rulebook`
静态规则定义。包含玩法、阶段、动作、公式、奖励规则。

2. `mini_game_session`
单局运行态。包含当前轮次、资源、参与者、公开态、隐藏态。

3. `mini_game_action_log`
每一步用户动作、系统判定、随机数消费记录。

4. `mini_game_writeback`
结算后回写主线的变更包，只允许写白名单字段。

### 3.2 状态边界
小游戏控制器只允许处理以下内容：

1. 小游戏内部的轮次与阶段。
2. 局内身份、资源、行动与结算。
3. 小游戏奖励与惩罚的结构化回写。
4. 小游戏结果摘要写入主线记忆。

小游戏控制器不允许处理以下内容：

1. 改写主线剧情结构。
2. 提前泄漏未解锁章节、角色秘密、后续事件。
3. 越权生成新主线任务。
4. 越权修改角色基础设定卡。
5. 把小游戏中的隐藏态直接说给用户。

## 4. 通用运行模型
### 4.1 建议状态枚举
`status`：

1. `idle`
2. `preparing`
3. `active`
4. `settling`
5. `finished`
6. `aborted`
7. `suspended`

`phase` 由具体小游戏定义，但必须存在。

### 4.3 公开态与隐藏态
`public_state` 给用户和前端看，`hidden_state` 只给控制器和后端看。

推荐约束：

1. 身份类秘密放 `hidden_state`。
2. NPC 决策倾向放 `hidden_state`。
3. 预生成随机序列放 `rng_state`。
4. 用户可感知到的资源放 `public_state`。
5. 奖励预览只显示范围，不显示隐藏结果。

### 4.4 随机数机制
为防止模型每轮胡乱“看感觉”判定，建议由后端预先给一个 `rng_queue`。

规则：

1. 每次需要随机判定时，控制器只消费队首或按 `cursor` 取值。
2. 所有公式都写在规则卡中，结果可复算。
3. `action_log` 记录本次用了哪个随机值。
4. 回放时可以按同一 `seed` 复现。

## 5. 通用流程
### 5.1 进入流程
1. 判断触发来源：主动输入标签或主线事件触发。
2. 检查进入条件：场景、角色、道具、体力、时间段。
3. 创建 `mini_game_session`。
4. 冻结主线快照摘要，记录本局可写回白名单。
5. 控制器返回开场说明、当前状态和第一批可选动作。

### 5.2 局内流程
每一轮固定顺序如下：

1. 读取当前 `phase` 和 `public_state`。
2. 用户输入动作，系统做动作归一化。
3. 控制器根据规则卡和随机数计算结果。
4. 产出 `state_delta`。
5. 应用 `delta` 后写入 session snapshot。
6. 判断是否换阶段、换轮次或结算。
7. 返回叙述文案、UI 状态和下一轮动作。

### 5.3 结算流程
1. 进入 `settling`。
2. 计算胜负、品质、产出、损耗、关系变化。
3. 生成 `writeback_delta`。
4. 只回写白名单字段。
5. 写入一条 `mini_game_summary` 到中期记忆。
6. 把 `status` 改为 `finished`。

### 5.4 暂停、恢复、退出
建议统一支持以下局内指令：

1. `查看状态`
2. `查看规则`
3. `继续`
4. `申请退出`
5. `确认退出`
6. `恢复小游戏`

协议：

1. `active` 状态下，若用户输入与局内动作无关的主线内容，控制器不进入主线，而是提醒“当前仍在小游戏中”。
2. `申请退出` 先进入确认，不直接丢局。
3. `suspended` 恢复后必须先返回局面摘要，再继续可选动作。
4. 被动触发的强制局一般不允许中途无损退出。

## 6. 动作归一化与校验
### 6.1 输入规范
每轮建议同时支持两种输入：

1. UI 按钮动作：最稳定，直接传 `action_id`。
2. 自然语言动作：由控制器映射到最近的合法动作。

### 6.2 合法性校验
如果用户输入不合法，控制器必须：

1. 不推进轮次。
2. 不改动局内状态。
3. 明确告知当前合法动作列表。

### 6.3 输出规范
每轮输出至少包含：

1. 当前 `game_type`
2. 当前 `phase`
3. 当前 `round`
4. 本轮结果摘要
5. 更新后的 `public_state`
6. 下一步合法动作 `player_options`
7. `state_delta`
8. 若结束则包含 `writeback_delta`

## 7. 主线回写白名单
小游戏结算后允许回写以下字段：

1. `player_state.resources`
2. `player_state.inventory`
3. `player_state.action_state`
4. `player_state.flags`
5. `relationship_state`
6. `progress.side_quests`
7. `event_pool.done`
8. `event_pool.failed`
9. `memory_state.mid_term`

默认不允许回写：

1. `current_chapter_id`
2. `current_scene_id`
3. 未触发的主线剧情信息
4. 新增未铺垫的重要世界设定
5. 角色基础参数卡原始值

## 8. 小游戏统一奖励规则
建议奖励由 3 层组成：

1. 固定奖励
例如经验、少量货币、局内消耗返还。

2. 表现奖励
根据评分、品质、是否无伤、是否首局通关追加。

3. 场景奖励
由当前章节或地点注入。例如烧烤局狼人杀后关系升温、灵脉挖矿额外掉灵石。

统一评分建议：

1. `S`：超额完成，稀有奖励。
2. `A`：成功完成，标准奖励。
3. `B`：勉强完成，基础奖励。
4. `C`：失败或中断，安慰奖励或无奖励。

## 9. 七个小游戏详细设计
## 9.1 狼人杀
### 9.1.1 定位
适合社交局、营火局、宿舍局、聚会剧情。重点是关系变化、观察角色性格、制造支线冲突。

### 9.1.2 MVP 局型
推荐默认使用 `5人标准局`，因为最稳：

| 局型 | 角色配置 | 适用 |
| --- | --- | --- |
| 4人极速局 | 狼人1、预言家1、村民2 | 人数不足时 |
| 5人标准局 | 狼人1、预言家1、女巫1、村民2 | 默认推荐 |
| 6人扩展局 | 狼人2、预言家1、女巫1、猎人1、村民1 | 后续增强 |

### 9.1.3 核心参数
`public_state`：

1. `day_count`
2. `alive_list`
3. `eliminated_list`
4. `public_vote_history`
5. `discussion_order`
6. `last_night_result`

`hidden_state`：

1. `role_map`
2. `wolf_target`
3. `seer_checks`
4. `witch_save_used`
5. `witch_poison_used`
6. `npc_suspicion_matrix`
7. `npc_personality_weight`

NPC 参数建议：

1. `logic`: `30~80`
2. `aggressive`: `20~80`
3. `trust_player`: `-30~30`
4. `lie_skill`: `10~90`

### 9.1.4 流程
固定阶段：

1. `setup`
2. `night_wolf`
3. `night_seer`
4. `night_witch`
5. `day_announce`
6. `day_discussion`
7. `day_vote`
8. `settling`

顺序说明：

1. 开局分配身份，身份只写 `hidden_state.role_map`。
2. 夜晚狼人选击杀目标。
3. 预言家查验一人阵营。
4. 女巫按需救人或毒人，每种药一局一次。
5. 白天公布昨夜死亡结果。
6. 白天每个存活角色发言一轮。
7. 所有人投票，平票则只在平票者中复投一次。
8. 第二次仍平票则本轮无人出局。

### 9.1.5 用户可选动作
按身份动态生成：

1. 狼人：`击杀X`、`空刀`、`查看局势`
2. 预言家：`查验X`、`跳过`
3. 女巫：`救X`、`毒X`、`双跳过`
4. 白天：`发言`、`投票X`、`弃票`、`查看记录`

### 9.1.6 胜负条件
1. 所有狼人出局，村民阵营胜。
2. 存活狼人数量大于等于其他存活人数，狼人阵营胜。
3. 若用户提前出局，允许进入旁观模式，直到本局结束。

### 9.1.7 结算与回写
回写重点：

1. 关系变化：根据发言、投票和是否“站队”调整角色好感与信任。
2. 事件摘要：记录“谁在狼人杀里表现异常”。
3. 奖励建议：胜利 `经验+30`，失败 `经验+10`，表现分追加 `关系+1~5`。

## 9.2 钓鱼
### 9.2.1 定位
轻度资源玩法，适合放松、采集、支线素材获取，也可以挂接烹饪、交易和炼药材料。

### 9.2.2 场景模板
| 场景 | 遇鱼率 | 稀有率 | 推荐鱼种 |
| --- | --- | --- | --- |
| 小河 | 55 | 5 | 鲫鱼、河虾 |
| 湖泊 | 45 | 12 | 青鱼、银鲤 |
| 深潭 | 35 | 20 | 灵鱼、古鳞鱼 |

### 9.2.3 核心参数
`public_state`：

1. `site_name`
2. `bait_name`
3. `stamina`
4. `focus`
5. `line_tension`
6. `fish_distance`
7. `hook_state`

`hidden_state`：

1. `target_fish_id`
2. `fish_stamina`
3. `fish_power`
4. `fish_escape_rate`
5. `encounter_roll`

### 9.2.4 基础公式
咬钩概率：

`hook_chance = 场景遇鱼率 + 鱼饵加成 + 时间段加成 + 装备加成 + 天气加成 - 噪音惩罚`

建议限制在 `10~95`。

中鱼后使用以下阈值：

1. `line_tension >= 100`，断线失败。
2. `fish_distance > 120`，脱钩失败。
3. `fish_distance <= 0 且 fish_stamina <= 0`，成功上岸。

### 9.2.5 流程
固定阶段：

1. `prepare`
2. `cast`
3. `waiting`
4. `hook_fight`
5. `landing`
6. `settling`

### 9.2.6 用户动作
1. `抛竿`
2. `提竿`
3. `收线`
4. `放线`
5. `稳杆`
6. `侧拉`
7. `强拉`
8. `换饵`
9. `收杆结束`

动作建议数值：

1. `收线`：`distance -20`，`tension +20`，`fish_stamina -10`
2. `放线`：`distance +10`，`tension -25`
3. `稳杆`：`distance -5`，`tension -15`，`fish_stamina -5`
4. `侧拉`：`distance -15`，`tension +10`，`fish_stamina -15`
5. `强拉`：`distance -30`，`tension +35`，`fish_stamina -25`

### 9.2.7 结算与回写
1. 鱼获写入 `inventory`。
2. 稀有鱼可附带 `event_pool.done` 事件。
3. 失败只扣鱼饵、体力和少量耐久，不做重罚。

## 9.3 修炼
### 9.3.1 定位
角色成长型玩法。目标不是“聊天修仙”，而是把修炼过程拆成可判定的资源管理与突破过程。

### 9.3.2 核心参数
`public_state`：

1. `qi`
2. `insight`
3. `stability`
4. `fatigue`
5. `breakthrough_progress`
6. `current_method`

`hidden_state`：

1. `deviation_risk`
2. `bonus_event_roll`
3. `environment_bonus`
4. `bottleneck_level`

参数范围建议：

1. `qi`：`0~100`
2. `insight`：`0~100`
3. `stability`：`0~100`
4. `fatigue`：`0~100`
5. `deviation_risk`：`0~100`
6. `breakthrough_progress`：`0~100`

### 9.3.3 流程
固定阶段：

1. `choose_method`
2. `gather_qi`
3. `circulate`
4. `breakthrough`
5. `settling`

### 9.3.4 用户动作
1. `吐纳`
2. `观想`
3. `稳息`
4. `服丹`
5. `冲关`
6. `收功`

动作建议数值：

1. `吐纳`：`qi +18`，`fatigue +8`
2. `观想`：`insight +15`，`fatigue +6`
3. `稳息`：`stability +18`，`deviation_risk -12`
4. `服丹`：`qi +25`，`stability -5`，每局最多一次
5. `冲关`：若 `qi >= 60`、`insight >= 40`、`stability >= 50`，则 `progress +25~45`，否则触发反噬
6. `收功`：安全结束，按当前进度折算经验

风控规则：

1. `fatigue > 70` 时，每轮 `deviation_risk +15`
2. `stability < 30` 时，`冲关` 失败概率大增
3. `deviation_risk >= 100` 立即中断本局，判定走火入魔失败

### 9.3.5 胜负条件
1. `breakthrough_progress >= 100` 且 `deviation_risk < 60`，突破成功。
2. 主动 `收功`，按当前积累发放普通收益。
3. 风险爆表则失败，写入负面状态或短期 debuff。

### 9.3.6 结算与回写
1. 回写修炼经验、境界进度、临时状态。
2. 成功突破可回写 `player_state.flags` 和 `skill_unlocks`。
3. 失败不直接改主线章节，只回写疲劳和状态异常。

## 9.4 研发技能
### 9.4.1 定位
用于“自创招式”“研发装置”“写法术模型”“技能迭代”。这是长期成长玩法，产出是技能蓝图、技能碎片、技能升级进度。

### 9.4.2 核心参数
`public_state`：

1. `target_skill_name`
2. `complexity`
3. `theory_progress`
4. `prototype_progress`
5. `stability_score`
6. `turn_left`

`hidden_state`：

1. `inspiration_roll`
2. `failure_threshold`
3. `synergy_bonus`
4. `mentor_bonus`

复杂度建议：

1. `1`：小技能修订
2. `2`：基础技能开发
3. `3`：进阶技能开发
4. `4`：稀有技能开发
5. `5`：特殊传承技能开发

### 9.4.3 流程
固定阶段：

1. `select_blueprint`
2. `theory_research`
3. `prototype_test`
4. `debug_refine`
5. `finalize`
6. `settling`

### 9.4.4 用户动作
1. `理论推演`
2. `样本拆解`
3. `原型实验`
4. `故障排查`
5. `请教同伴`
6. `强行迭代`
7. `定稿封存`

动作建议数值：

1. `理论推演`：`theory +20`
2. `样本拆解`：`theory +15`，消耗材料 `1`
3. `原型实验`：`prototype +25`，`stability -10`
4. `故障排查`：`stability +20`
5. `请教同伴`：`theory +10` 或 `inspiration +15`
6. `强行迭代`：`prototype +35`，`stability -20`
7. `定稿封存`：达标后进入结算

完成阈值建议：

1. `theory_progress >= 60 + complexity * 10`
2. `prototype_progress >= 50 + complexity * 10`
3. `stability_score >= 40`

### 9.4.5 结算与回写
1. 成功：回写 `skill_unlocks`、`skill_book`、`event_pool.done`
2. 半成功：回写 `skill_fragment`
3. 失败：只保留研究笔记，不解锁正式技能

## 9.5 炼药
### 9.5.1 定位
资源转化型玩法，适合战斗前准备、治疗、强化、宗门生活类剧情。

### 9.5.2 核心参数
`public_state`：

1. `recipe_name`
2. `round_limit`
3. `heat`
4. `purity`
5. `fusion`
6. `toxicity`
7. `stability`

`hidden_state`：

1. `target_heat_min`
2. `target_heat_max`
3. `required_order`
4. `ingredient_match_score`
5. `rare_mutation_roll`

数值范围建议：

1. `heat`：`0~100`
2. `purity`：`0~100`
3. `fusion`：`0~100`
4. `toxicity`：`0~100`
5. `stability`：`0~100`

### 9.5.3 流程
固定阶段：

1. `select_recipe`
2. `preheat`
3. `main_refine`
4. `stabilize`
5. `condense`
6. `appraise`
7. `settling`

### 9.5.4 用户动作
1. `升火`
2. `降火`
3. `投药`
4. `搅拌`
5. `稳炉`
6. `提纯`
7. `凝丹`

动作建议数值：

1. `升火`：`heat +15`
2. `降火`：`heat -15`
3. `投药`：顺序正确则 `fusion +20`，错误则 `toxicity +15`
4. `搅拌`：`fusion +10`，`stability +5`
5. `稳炉`：`stability +15`，`heat -5`
6. `提纯`：`purity +15`，`fusion -5`
7. `凝丹`：满足阈值时进入结算

风险规则：

1. 热度超出目标带 `20` 以上时，`stability -10`，`toxicity +10`
2. `toxicity >= 100` 判定炸炉失败
3. `stability <= 0` 判定药液报废

品质公式建议：

`quality_score = purity * 0.35 + fusion * 0.35 + stability * 0.2 - toxicity * 0.2 + 配方匹配加分`

### 9.5.5 结算与回写
1. 回写成品药品到 `inventory`
2. 高品质丹药可额外回写 `event_pool.done`
3. 失败时只扣材料与少量体力，不建议连带大面积惩罚

## 9.6 挖矿
### 9.6.1 定位
采集与探索型玩法，适合资源产出、危险探索和章节素材积累。

### 9.6.2 场景模板
| 场景 | 富矿度 | 风险值 | 稀有率 |
| --- | --- | --- | --- |
| 露天矿 | 60 | 20 | 5 |
| 洞窟矿 | 80 | 40 | 12 |
| 灵脉矿 | 100 | 60 | 20 |

### 9.6.3 核心参数
`public_state`：

1. `mine_name`
2. `vein_hp`
3. `stability`
4. `danger`
5. `player_stamina`
6. `tool_durability`
7. `bag_load`

`hidden_state`：

1. `ore_table`
2. `rare_drop_roll`
3. `weakness_point`
4. `collapse_threshold`

### 9.6.4 流程
固定阶段：

1. `survey`
2. `choose_vein`
3. `excavate`
4. `risk_check`
5. `haul`
6. `settling`

### 9.6.5 用户动作
1. `勘探`
2. `开采`
3. `精挖`
4. `支护`
5. `清障`
6. `休息`
7. `撤离`

动作建议数值：

1. `勘探`：发现弱点后，下次 `开采效率 +20%`
2. `开采`：`vein_hp - 基础采矿力`，`stamina -15`，`danger +10`
3. `精挖`：小幅削减 `vein_hp`，`稀有率 +15%`，`stamina -20`
4. `支护`：`stability +20`，`danger -15`
5. `清障`：`bag_load -10` 或提升下一轮效率
6. `休息`：`stamina +20`
7. `撤离`：保留当前收益并结算

风险规则：

1. `danger >= 70` 时，每轮都要做坍塌判定。
2. `stability <= 20` 且继续开采时，坍塌概率大增。
3. 坍塌后优先扣体力、耐久和部分矿石，不直接写死剧情失败。

### 9.6.6 结算与回写
1. 矿石、宝石、灵石写入 `inventory`
2. 受伤或疲劳写入 `player_state.resources` 或 `flags`
3. 若挖到剧情关键矿，可额外写 `event_pool.done`

## 9.7 升级装备
### 9.7.1 定位
这是制造和强化玩法，不建议做成单纯“点一下看运气”。更推荐做成短回合锻造，让用户通过操作降低失败风险。

### 9.7.2 核心参数
`public_state`：

1. `equip_id`
2. `equip_name`
3. `equip_type`
4. `current_level`
5. `target_level`
6. `forge_heat`
7. `forge_progress`
8. `stability`
9. `spirit_sync`

`hidden_state`：

1. `optimal_heat_min`
2. `optimal_heat_max`
3. `failure_risk`
4. `bonus_affix_roll`

推荐热区：

1. 武器：`60~80`
2. 护甲：`50~70`
3. 饰品：`40~60`

### 9.7.3 流程
固定阶段：

1. `select_equip`
2. `heat_up`
3. `forge`
4. `temper`
5. `infuse`
6. `finalize`
7. `settling`

### 9.7.4 用户动作
1. `加热`
2. `锻打`
3. `校正`
4. `淬火`
5. `注灵`
6. `收工定型`

动作建议数值：

1. `加热`：`heat +20`，`progress +5`
2. `锻打`：热区正确时 `progress +20`，否则 `stability -10`
3. `校正`：`stability +15`，`heat -10`
4. `淬火`：`spirit_sync +10~25`，错误淬火则 `stability -15`
5. `注灵`：`spirit_sync +20`，`failure_risk +10`
6. `收工定型`：满足阈值后进入结算

成功阈值建议：

1. `forge_progress >= 80`
2. `stability >= 50`
3. `spirit_sync >= 40`

### 9.7.5 结果分级
1. 完美成功：`+1` 强化等级，并追加一个小词条或额外耐久。
2. 普通成功：`+1` 强化等级。
3. 半成功：等级不变，但提升耐久或积累保底值。
4. 失败：扣材料与耐久。

MVP 不建议默认掉级，避免用户体验过差。若后续要加掉级，只建议在 `+6` 以上且高风险操作时开放。

### 9.7.6 结算与回写
1. 回写装备当前等级、耐久和附加词条。
2. 消耗材料从 `inventory` 扣除。
3. 结果写入中期记忆，便于后续剧情引用“这件武器是谁锻出来的”。

## 10. 数据落地建议
在已有 `game_run` 体系上，建议最少新增 3 张表或 3 类对象：

1. `game_run_mini_game_session`
字段：`id, game_run_id, game_type, status, phase, round, session_json, created_at, updated_at`

2. `game_run_mini_game_action`
字段：`id, session_id, round, actor_id, action_id, action_payload_json, rng_used, result_json, created_at`

3. `game_run_mini_game_result`
字段：`id, session_id, result, grade, reward_json, writeback_json, summary_text, created_at`

如果暂时不想扩表，也可以先把这三层放入 `snapshot_json`，但长期还是建议拆开。

## 11. 与主线 Agent 的协作协议
### 11.1 主线进入小游戏
主线编排师只负责：

1. 判定“是否进入小游戏”
2. 提供入口上下文
3. 暂停主线推进

然后把控制权交给 `mini_game_agent`。

### 11.2 小游戏结束后
小游戏控制器只返回：

1. `mini_game_result`
2. `writeback_delta`
3. `memory_summary`

主线编排师收到后再决定如何把结果体现在剧情里。

## 12. 交付结论
建议不要再使用单句版提示词直接跑 7 个小游戏。更稳的做法是：

1. 用一个通用 `mini_game_controller` 管入口、状态机、回写边界。
2. 每个小游戏配一张独立 `rulebook`，把阶段、动作、公式和奖励写死。
3. 通过 `public_state + hidden_state + rng_queue + writeback_whitelist` 保证稳定性。
4. 先按本文的 MVP 参数落地，再逐步扩展更多局型、鱼种、丹方、矿脉和强化路线。
